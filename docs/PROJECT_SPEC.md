# Repo2Prompts Project Specification

## 1. Product Summary

Repo2Prompts is an evidence-backed repository intelligence tool for AI coding agents.

Users paste a public GitHub repository, choose an analysis mode and depth, and receive an agent-ready technical brief grounded in selected repository evidence. The product is designed to be more useful than plain repo packers: it selects relevant files, skips unsafe or low-value files, cites evidence paths, tracks analysis quality, and produces shareable reports.

Core positioning:

> Not just repo packing. Evidence-backed briefs for AI coding agents.

Primary users:

- Developers using Codex, Cursor, Claude, or other AI coding agents.
- Freelancers and agencies preparing technical handoff or review reports.
- Teams that need fast onboarding, review, debug, or migration briefs for public repositories.

Current monetization posture:

- Product quality first.
- Ads and analytics are deferred.
- No payment or paywall exists in the current app.

## 2. Core Value Proposition

Repo2Prompts turns repository source evidence into structured briefs that coding agents can act on.

Key differentiators:

- **Evidence selection:** the app chooses files instead of dumping the whole repository.
- **Citations:** important claims cite repository paths such as `[src/app.ts]`.
- **Task modes:** build, review, debug, migration, and prompt outputs use different structures.
- **Adaptive depth:** fast, balanced, focused, and deep flows trade latency for analysis depth.
- **Quality metadata:** warnings and repair status are exposed instead of hidden.
- **Shareable reports:** stored briefs can be viewed as report pages.
- **Provider-neutral runtime:** any OpenAI-compatible model gateway can be used.

## 3. User-Facing Features

### 3.1 Workbench

Route:

```text
/
```

The workbench lets users:

- enter `owner/repo` or a GitHub URL,
- choose analysis mode,
- choose analysis depth,
- optionally enter a focused question,
- optionally add include/exclude file filters,
- generate an evidence-backed brief,
- inspect selected/skipped/largest files,
- inspect analysis metadata and quality warnings,
- copy/download/export the result.

Sample repositories are shown on the first screen. The homepage also links to static examples.

### 3.2 Analysis Modes

Supported modes:

```ts
type AnalysisMode = "build" | "review" | "debug" | "migration" | "prompt";
```

Mode behavior:

- `build`: product purpose, architecture, implementation plan, verification plan.
- `review`: architecture summary, risks, maintainability, missing tests, review checklist.
- `debug`: runtime path summary, likely failures, diagnostics, minimal fixes.
- `migration`: current architecture, migration target, phased plan, rollback.
- `prompt`: concise agent prompt, constraints, unknowns.

Each mode has required Markdown heading contracts in `src/domain/brief-prompt.ts`.

### 3.3 Analysis Depths

Supported depths:

```ts
type AnalysisDepth = "fast" | "balanced" | "deep" | "focused";
```

Depth behavior:

- `fast`: one writer call from selected evidence; lowest latency.
- `balanced`: creates or reuses a repository map, then writes the brief.
- `focused`: ranks evidence using a user question and uses a question-specific repository map.
- `deep`: groups files into modules, analyzes bounded modules, merges maps, writes the final brief, and may repair once.

Backward compatibility:

- request depth `thorough` maps to `balanced`.

### 3.4 Deep Progress UX

The workbench displays timer-based loading messages:

- preparing request,
- collecting repository evidence,
- selecting files,
- building repository map or module map,
- writing brief,
- long-running fallback messages after 30s and 75s.

This is client-side only. The API is not streamed in the current version.

### 3.5 Shareable Reports

Routes:

```text
/library
/library/[id]
```

The library lists stored briefs from the optional database. The report detail page presents:

- repository, mode, depth, date, and views,
- summary stats,
- readable Markdown brief,
- selected evidence,
- skipped files,
- largest files,
- pipeline and quality metadata,
- copy/download/export/share actions.

Old stored rows with missing metadata render gracefully.

### 3.6 Static Examples

Routes:

```text
/examples
/examples/[slug]
```

Static examples are available without model or database configuration:

- Next.js build brief,
- Supabase review brief,
- GitHub Action debug brief,
- CLI migration brief.

Examples use the same report UI as library detail pages.

### 3.7 Export Actions

The action bar supports:

- copy Markdown,
- download Markdown,
- copy Codex handoff,
- copy Cursor brief,
- copy Claude brief,
- export evidence JSON,
- copy share link.

Export formats are generated by `src/domain/brief-export.ts`.

Exports include:

- repository,
- mode,
- depth,
- selected evidence paths,
- skipped evidence paths,
- quality warnings,
- brief text.

Exports do not include:

- full source content,
- internal prompts,
- model API keys,
- database keys.

## 4. API Surface

### 4.1 Generate/List Briefs

Route:

```text
/api/briefs
```

Methods:

- `GET`: list stored briefs from the optional database.
- `POST`: generate a new brief or return an evidence-fingerprint cache hit.

Request body:

```ts
{
  repository: string;
  mode?: "build" | "review" | "debug" | "migration" | "prompt";
  depth?: "fast" | "balanced" | "deep" | "focused";
  question?: string;
  include?: string;
  exclude?: string;
}
```

Response shape:

```ts
{
  brief: string;
  source: "generated" | "cache";
  mode: AnalysisMode;
  depth: AnalysisDepth;
  evidence: StoredEvidence;
  analysis: AnalysisMetadata;
  reportId?: string;
  reportUrl?: string;
}
```

Typed error shape:

```ts
{
  error: {
    code: string;
    message: string;
    requestId: string;
  }
}
```

Known error codes:

- `RATE_LIMITED`
- `INVALID_REQUEST`
- `INVALID_REPOSITORY`
- `QUESTION_REQUIRED`
- `REPOSITORY_NOT_FOUND`
- `GITHUB_UNAVAILABLE`
- `INVALID_FILTER`
- `NO_EVIDENCE`
- `MODEL_UNAVAILABLE`
- `INTERNAL_ERROR`

### 4.2 Status

Route:

```text
/api/status
```

Returns configured/disabled state for:

- GitHub token,
- chat model,
- embeddings,
- database,
- rate limit backend.

It does not expose secrets or model API keys.

## 5. Data Model

### 5.1 Stored Briefs

Table:

```text
repository_briefs
```

Purpose:

- durable cache,
- shareable library reports,
- evidence metadata storage.

Important fields:

- `repository_key`
- `analysis_mode`
- `analysis_depth`
- `question_hash`
- `evidence_fingerprint`
- `title`
- `brief_markdown`
- `evidence_json`
- `view_count`
- timestamps

Cache identity:

```text
repository_key + analysis_mode + analysis_depth + question_hash + evidence_fingerprint
```

This prevents stale cache reuse when source evidence changes.

### 5.2 Repository Analysis Cache

Table:

```text
repository_analysis_cache
```

Purpose:

- cache repository maps for balanced/focused/deep flows.

Important fields:

- `repository_key`
- `resolved_ref`
- `repository_path`
- `evidence_fingerprint`
- `scope`
- `question_hash`
- `repository_map`
- `evidence_json`

RLS is enabled. No direct public policies are intended.

### 5.3 Rate Limit Events

Table:

```text
rate_limit_events
```

Purpose:

- optional shared database-backed rate limit for public multi-instance deployment.

Important fields:

- `key_hash`
- `route`
- `created_at`

The table stores hashed request keys, not raw IP addresses.

## 6. Analysis Pipeline

Implementation:

```text
src/integrations/analysis-pipeline.ts
```

Flow:

1. Parse repository input.
2. Create cache identity.
3. Collect repository evidence from GitHub.
4. Compute evidence fingerprint.
5. Check stored final brief cache.
6. If not cached:
   - fast: write directly from evidence.
   - balanced/focused: create or reuse repository map, then write.
   - deep: group files into modules, analyze modules, merge maps, then write.
7. Validate brief quality.
8. Repair once when allowed.
9. Save final brief if database is configured.
10. Return brief, evidence, analysis metadata, and optional report link.

Failure behavior:

- no evidence: typed API error,
- model unavailable: typed API error,
- database failures: generation continues without cache,
- bad analyst JSON: deterministic fallback map,
- module failures: fallback maps can keep deep generation moving.

## 7. Evidence Collection

Implementation:

```text
src/integrations/github-reader.ts
src/domain/context-policy.ts
src/domain/file-analysis.ts
src/domain/relationship-graph.ts
src/domain/module-grouping.ts
```

Evidence behavior:

- accepts owner/repo and GitHub URLs,
- supports branch/path URLs,
- reads metadata, tree, and selected files,
- ranks files based on role, mode, question, graph signals, and diversity,
- skips generated/oversized/unsafe files,
- records skipped reasons,
- estimates tokens,
- keeps selected and largest file summaries.

Security guard:

- skips obvious secret paths,
- skips token-looking content,
- include filters cannot force secret-looking files into model context.

## 8. Model Provider Runtime

Implementation:

```text
src/integrations/model-client.ts
src/integrations/chat-completion.ts
```

Provider shape:

- OpenAI-compatible `/chat/completions`.
- Supports JSON and SSE-style completion extraction.
- Uses `MODEL_BASE_URL`, `MODEL_API_KEY`, `MODEL_CHAT_ID`.
- Uses task-specific model overrides:
  - `MODEL_ANALYSIS_ID`
  - `MODEL_WRITER_ID`

Provider presets documented:

- 9router,
- Groq,
- OpenRouter,
- Gemini-compatible gateway.

Native Gemini SDK, embeddings, Jina, and Jira are not part of the current implementation.

## 9. Rate Limiting And Logging

### Rate Limit

Implementation:

```text
src/integrations/rate-limit.ts
```

Default:

- in-memory limiter,
- `10 requests / 10 minutes`.

Optional:

- database limiter through `RATE_LIMIT_BACKEND=database`.

Environment:

```env
RATE_LIMIT_BACKEND=memory
RATE_LIMIT_DB_FAILURE_MODE=open
RATE_LIMIT_WINDOW_MS=600000
RATE_LIMIT_MAX=10
```

### Logging

Implementation:

```text
src/integrations/server-log.ts
```

Logs safe JSON events with:

- route,
- request id,
- event,
- error code,
- repository,
- mode,
- depth,
- duration,
- source.

Logs never include:

- API keys,
- service keys,
- full source content,
- full prompts,
- full generated briefs.

## 13. Current Limitations

- Only public GitHub repositories are supported.
- No native private repo authentication flow in the browser.
- No native Gemini API.
- No embedding-based semantic search.
- No ads implementation yet.
- No payment/subscription system.
- No full Markdown library; renderer is intentionally lightweight.
- Deep mode can still be slow for larger repositories.
- Database-backed rate limit migration exists but must be applied before production use.
